Why Superman

它解决什么问题

真实 Agent 系统的问题通常不是“模型不够聪明”，而是上下文、记忆、工具、权限和协作边界没有被工程化。Superman 的设计重点是把这些长期会拖垮系统的问题显式拆开。

设计前提

Agent 要进入真实工作流，必须能长期保存状态、能解释自己做过什么、能把历史经验变成更好的下一次执行，还要能在需要时把任务交给更合适的专家。Superman 的很多设计都围绕这些工程问题展开。

上下文污染

长上下文并不天然可靠。Superman 使用持久会话、精简演化日志、文件 revision tracking 和自动压缩，把可用信息留住，把噪声隔离出去。

能力不可复用

普通 Agent 做完任务就结束。Superman 会把完成的工作沉淀到 L1 全局事实、L2 SOP、专家记忆和专家系统提示中，让下一次任务更快进入状态。

单体 Agent 难维护

Superman 通过专家委托把领域能力拆到独立专家目录，每个专家拥有自己的 soul、记忆、会话和运行边界，主 Agent 负责协调而不是吞下所有职责。

自动化缺少治理

运行时事件写入 JSONL audit log，Hook 可以在模型、工具、Agent、run 等关键节点前后介入，适合做审批、记录、拦截、通知和安全策略。

问题一：历史越多，越难用

如果把所有历史直接塞进上下文，Agent 很快会被过期信息、重复过程和临时细节淹没。Superman 用 SQLite 保存完整消息，用 compact log 和自动压缩保留更适合进化的投影，让“可追溯”和“可继续使用”分开处理。

问题二：文件改动不可控

自治 Agent 一旦能写文件，就必须能追踪自己改了什么。Superman 的 session 文件追踪会保存 revision snapshots，并提供 history、diff、revert 等命令，减少“改了但说不清”的风险。

问题三：工具越接越乱

内建工具、MCP 工具和 Skill 能力如果没有统一装配，会让 Agent 的可用能力难以判断。Superman 通过配置和 sm toolsets 让当前工具集合可见，便于调试和治理。

问题四：后台学习容易漂移

自进化如果没有范围限制，会把错误经验写进核心行为。Superman 把 agent evolution 和 meta evolution 分层：前者优化主 Agent 与专家，后者只优化进化过程本身。

风险Superman 的处理方式
上下文过长持久会话、自动 compact、精简演化日志。
记忆污染L1 全局事实、L2 SOP、专家记忆、evolver 记忆分离。
工具不可审计runtime events JSONL、Hook 生命周期节点、工具错误事件。
专家职责混杂专家以目录组织，soul.md、memory、state.db、sessions 独立保存。
