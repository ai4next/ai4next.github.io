Workflow

Superman 如何工作

  1. 接收任务：用户可以通过 TUI、sm run、prompt 文件或 IM 平台提交任务。
  2. 构建上下文：系统加载配置、模型、工具、Skill、MCP toolset、记忆索引和相关会话上下文。
  3. 执行与协作：主 Agent 根据任务调用工具，必要时委托专家子 Agent 独立执行并返回结果。
  4. 记录与审计：工具调用、文本增量、错误、模型事件和进化事件持续写入 runtime audit log。
  5. 沉淀与进化：任务完成后，后台流程把经验压缩到记忆、SOP、专家能力或 evolver 记忆中。

三种常用入口

入口适合场景示例
交互式 TUI连续对话、探索代码库、逐步推进复杂任务。sm
一次性执行脚本、自动化任务、CI 辅助检查、命令行问答。sm run "提示词"
即时通信服务团队聊天入口、机器人、群内自动化任务。sm im serve

一次任务的生命周期

Pre-run

加载配置，创建 runtime 目录，初始化模型 provider、内建工具、MCP server、skills、hooks 和会话上下文。

Run

主 Agent 根据当前任务和上下文决策是否调用工具、读取文件、写入文件、执行命令、询问用户或委托专家。

Audit

运行中产生的工具调用、文本增量、错误和进化事件会流式写入 JSONL 审计日志，方便后续查询和统计。

Post-run

后台进行归档、会话压缩、记忆沉淀、专家培养和分层进化，尽量不阻塞当前用户任务。

自主反思与调度

sm reflect 会启动自主空闲监听和调度模式。配置中的 reflect.autonomous.idle_timeout 控制空闲多久后进入自主反思，reflect.scheduler.tasks_dir 指向任务定义目录，适合定期检查、健康巡检和后台改进任务。

工作方式重点：用户不应该为系统整理历史、压缩上下文和培养专家付出额外等待时间。Superman 倾向把这些维护动作放在 run 之后的后台阶段。
