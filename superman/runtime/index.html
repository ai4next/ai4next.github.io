Runtime

运行数据与治理边界

Superman 默认将运行时数据放在 ~/.sm/。其中 state.db 保存会话与消息元数据，sessions/ 保存精简日志和文件快照，runtime/events.jsonl 保存审计事件，memory/ 保存主 Agent 的 L1/L2 记忆，experts/ 保存每个专家的 soul、记忆和会话，evolution/ 保存进化系统自己的记忆与会话。

这种目录结构让用户可以直接检查、备份、迁移或删除某类数据，也方便把不同责任域接入不同的治理策略。

Workspace 目录说明

路径说明
config.yaml用户配置，由 sm init 或 sm configure 创建。
state.db主 Agent 的 SQLite session/message 元数据和完整消息。
sessions/精简会话日志、文件 revision snapshots 和会话相关投影。
runtime/events.jsonl运行时审计事件，包括工具调用、文本增量、错误和进化事件。
memory/主 Agent 扁平文件记忆，包含 L1 facts 和 L2 SOP。
experts/专家目录，每个专家都有 soul、memory、state.db 和 sessions。
evolution/进化系统运行根目录，保存 evolver 自己的记忆、会话和日志。
hooks/ / skills/用户自定义 Hook 脚本和 Skill 定义。

审计与查询

运行时事件不是只写给机器看的日志，也是一种治理接口。你可以用 Superman 自带命令查看 event 列表和 summary，用外部脚本读取 JSONL，也可以在 Hook 中把关键事件同步到组织的日志系统。

sm runtime events
sm runtime summary

会话治理

会话保存完整消息，也保存文件快照和精简投影。对于长期使用的 Agent，这让你可以回答三个问题：它当时为什么这么做，它改过哪些文件，如果错了能否回滚。

sm sessions list
sm sessions show <id>
sm sessions files <id>
sm sessions diff <id> <path>
sm sessions revert <id> <path>

循环检测与生命周期

配置中的 session.max_turns、archive_interval、session_ttl 和 loop_detection 用于控制长期会话行为。循环检测可以配置 window size 和 max repeats，减少 Agent 在同类动作中反复打转。

治理建议：生产化使用时，优先开启审计、限制 IM 来源、明确工具权限，并定期检查 session storage 与 runtime summary。
